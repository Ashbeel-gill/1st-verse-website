# 1st-verse — Landing Page

Marketing site for **1st-verse**: premium human CX (customer experience) agents for Shopify brands, SaaS platforms, and modern online businesses. Static site — no build step, no framework. Open `index.html` or serve the folder.

Owner: Ashbeel (ashbeel@helpthemallfoundation.org). Live domain: `1st-verse.com` (currently hosts the old receipt-themed React site; this project replaces it).

## Business facts (source of truth for copy)

- **Pricing model**: **published volume rates (no gated quotes)** — **$15/hr** for up to 40 hrs/wk, **$12/hr** for 40+ hrs/wk, **$10/hr** for 24/7 (120+ hrs); **multiple agents → book a call**. (Framed by **hours**, not agent-count — owner rejected the "2 agents" label; agent-count appears only as the book-a-call row.) All-in, billed to the minute, itemized in a weekly "Monday ledger." Framed on-site as "one simple rate that scales," deliberately *not* as "plans/tiers" — transparency vs. the field's gated pricing is the hook. **$15 stays the headline/anchor** everywhere (coin, hero, meta, OG); "from $15" only where honesty needs it. $10 is the floor — do not go lower.
- **Comparison baselines**: in-house agent at **$22/hr base wage + 40% overhead** (≈$31/hr loaded), and a **managed agency at $35/hr**. All three are researched (US CX wage ~$18–22; fully-loaded multiplier 1.3–1.5×; onshore BPO $28–45/hr) but still owner-to-confirm.
- **Calculator math**: `monthly = hours × 52 × rate ÷ 12`. The 1st-verse rate is a **step function of weekly hours** — `rateFor(h)` in `app.js`: `h>120 → $10` (`RATE_FV_247`), `h>40 → $12` (`RATE_FV_TEAM`), else `$15` (`RATE_FV_1`). Baselines `RATE_INHOUSE=22`, `OVERHEAD=0.4`, `RATE_AGENCY=35`. Slider is **5–168 hrs**; presets `10/20/40/80/24-7(=168)`. Savings = in-house total − 1st-verse. The ledger renders a **three-bar comparison** (agency / in-house / 1st-verse) scaled to the tallest (agency); the in-house & agency bars are constant-width, but the **1st-verse bar shrinks at higher tiers** (its rate drops while the others stay flat) — reinforcing "cheaper per hour at scale." The big rate number, a tier caption (`data-tier-label`: "up to 40 hrs / week" / "40+ hrs / week" / "24/7 coverage") and inline `$` labels (`data-fv-rate`, in the rate block + ledger row + fv bar) all update live with the slider.
- **Positioning**: 100% human agents, no bots ever (contractual), <60 min to launch, 7-day free trial, no contracts, no card required.
- **Process**: Connect (60s) → Train (brand voice, past tickets, macros; 1–3 days) → Launch (shared Slack channel) → Scale (Monday ledger).
- **Coverage**: follow-the-sun teams in Lisbon, Manila, Bogotá (24/7).
- **Booking**: Cal.com, calLink **`1st-verse/30min`**, brand color `#1972F5`.
- **Placeholder content that needs real data before launch**: 4.9★ CSAT, 38s first response, "97" QA score (feature card), "+$3,459 saved this month," the "+$184 cart rescued" chip, the illustrative Lisbon/Manila/Bogotá local-time clocks in the 24/7 feature card, and the $22/+40% in-house assumptions (≈$31/hr loaded, also surfaced in the comparison table and the pricing three-bar) plus the $35/hr managed-agency rate — all researched but owner to confirm.

## Design system — "Crisp DNA"

The visual direction is **crisp.chat's exact system**, extracted from their live CSS. Do not drift from it. Light mode only — **dark mode was built and deliberately removed; do not reintroduce it.**

Tokens (all in `:root` in `styles.css`):

- Blue `#1972F5` (`--blue`), hover `#1258D6`. The ONLY dominant accent.
- Navy text `#030D26` (`--navy`), slate body `#5D6B98`, `#404968`.
- Backgrounds: near-white `#FCFCFD` / `#F8F8FC` — never pure gray, never pastel page background.
- Cards: white, on an **ultra-soft shadow system** (Premium Polish sprint): a faint 1px ring drawn as a low-alpha navy `box-shadow` (not `border`, so no layout shift) plus a large soft drop. Tokens `--ring-shadow` / `--soft-shadow` / `--lift-shadow` / `--big-shadow`, all navy `rgba(3,13,38,·)` at 0.05–0.14. Heavier lift only on hover or hero-level cards. (This replaced the earlier crisp `#EAECF5` ring + whisper shadow.)
- Secondary colors (orange `#FF6F3E`, purple `#955CF4`, green `#2DC862`, yellow `#FFC53D`) appear ONLY in small icon tiles, doodles, and accent chips — never as section backgrounds except the two pastel showcases (`--show-blue`, `--show-purple`).
- Radii: 12px buttons, 16–24px cards, 32px showcase containers.
- Font: **Figtree** 400–900 (closest free match to Crisp's Aeonik Pro). Headlines 800–900, letter-spacing −0.03em; **section & showcase titles are weight 900** (Premium Polish). Body copy is deliberately small and muted (`--slate`) for high-end typographic contrast.
- Signature: the **marker highlight** (`.hl`) — light-blue rounded swipe behind one headline word, rotated −1.2°.
- **Grain** (`--grain`): a static SVG fractal-noise data-URI, layered as a `background-image` under `soft-light` blend on the blue surfaces — finale, primary buttons (`.btn--blue`), agent bubbles + typing indicator, chat header, chat launcher, and the theatre's Slack "you" bubble / Send button. (The pricing coin was in the original grain spec but left plain.)
- **Section texture**: `#services` sits on a subtle masked **dot-grid** (`::before`, same motif as hero/finale). This replaced an earlier blurred blue/purple "glow-blob" background that read as AI-generated.

Style guardrails (hard-won user feedback — violating these reads as "AI-generated"):
- One accent color; no rainbow gradients, no gradient text.
- Almost no emoji in UI copy (a couple inside chat bubbles is the ceiling).
- Monogram avatars (`.ava--sk` etc.), never smiley faces.
- No full app-dashboard mockups (a "workspace" hero was built and rejected as un-premium). Product story is told through the floating conversation card + chips.
- Marquee shows **CX platforms, never fake client names**.

## Animation architecture (`app.js`)

**Hard performance contract** (from the original brief — do not break):
- GSAP animates **only** transforms (`x/y/z/scale/rotation/rotationX/rotationY`) and `opacity`/`autoAlpha`. Never width/height/top/left. The chart reveal uses a scaleX curtain, not path/dashoffset. Card hover shadows animate via pre-rendered layers/box-shadow CSS transitions, not per-frame shadow tweens.
- `gsap.matchMedia()` with three contexts: desktop (≥768px) gets tilt/parallax/aurora/magnetic/glare/idle-floats; mobile gets lightweight opacity fades and static layout (chips become static elements, some hidden); `prefers-reduced-motion` gets a fully static page (CSS block at the bottom of `styles.css` + early return in JS).
- Event listeners registered inside contexts are cleaned up via `context.add(() => () => …)`.
- Idle floats use `yPercent` so they never fight parallax/tilt `x/y` (GSAP keeps the channels separate).
- Elements pre-hidden for choreography via `.js [data-reveal] / [data-stagger] / [data-convo-item] / [data-show-*] / [data-v-pop] / [data-feat-pop] / [data-cmp-pop]` CSS (the `js` class is added by app.js; no-JS users see everything).

Key systems, in file order: `countUp` utility → hero intro timeline → conversation loop → scroll reveals (singles) → `ScrollTrigger.batch` for grids → showcase scene timelines → **how-it-works theatre** (`initTheatre` — pinned, scrubbed master timeline + cumulative pipeline; the pin has `refreshPriority: 1` so it's measured before the triggers below it) → **feature micro-demos** (`initFeatureDemos` — one scroll-triggered demo per bento card) → **comparison table** (`initCompare` — row-by-row tick cascade) → aurora drift → hero scene 3D tilt → card 3D tilt + glare → idle floats → magnetic CTAs → mobile nav → FAQ expansion → **Cal.com fallback** (if embed.js fails to load, `[data-cal-link]` clicks open `cal.com/…` directly — detection via script error event + a 4s queue-drain check) → calculator → chat widget.

## Dev & verification

- Preview: `preview_start` with the `1st-verse` config (`.claude/launch.json`, `npx serve` on port 4173), or any static server.
- **Preview-panel quirks** (harness, not the site): the tab is throttled — GSAP lag-smoothing makes animations crawl and CSS transitions freeze, `requestAnimationFrame` starves entirely, and `preview_screenshot` intermittently hangs until the server is stopped/restarted. Judge motion only in a focused real browser tab. Prefer `preview_eval` + computed-style probes for verification.
- Emulated `preview_resize` does **not** fire matchMedia change events — reload after resizing to re-run GSAP contexts.
- The Cal.com embed is blocked in the sandboxed preview (`ERR_BLOCKED_BY_RESPONSE`, cross-origin isolation); it works in a normal browser. The JS fallback covers real-world failures.
- Breakpoints to check: 1280 (full), ~820 (tablet, burger appears ≤860), 375 (mobile). Always check `document.documentElement.scrollWidth > clientWidth` for overflow regressions — absolutely-positioned floaters are the usual culprit.
- Known footgun: scoping selectors like `.stat span` or `.ledger__row span` too loosely catches nested number spans — use `>` child combinators around the ledger/stats.
- **Token economy** (the transcript is the main drain — it re-processes every turn): edit via `Grep` → targeted `Read(offset/limit)` → `Edit`, never full-file re-reads; batch verification into a single `preview_eval` rather than many round-trips; and `/clear` (or a fresh session) between distinct work chunks — code + this file persist, so nothing is lost.
- Verifying **scroll-scrubbed motion** (theatre, feature demos) in the throttled preview: the GSAP ticker is starved, so nothing plays on its own. Drive it directly — `ScrollTrigger.getAll().filter(t=>t.pin)[0].animation.progress(p)` for the theatre — or force-complete tweens (`gsap.getTweensOf(el).forEach(t=>t.progress(1))`) and read computed styles. `preview_resize` reflows CSS grid/media queries immediately (bento breakpoints are pure CSS), but JS `matchMedia` contexts still need a reload.

## graphify

This project has a knowledge graph at graphify-out/ with god nodes, community structure, and cross-file relationships.

Rules:
- **Scope check first — graphify indexes `app.js` (AST) only; CSS and HTML are not in the graph.** Most work on this project is styling/markup, so for those tasks **skip the query entirely**: `Grep` for the exact selector/string, then `Read` with `offset`/`limit` — do **not** re-read the whole file (`styles.css` is ~1,500 lines; a full read is a large, repeated token cost). Reserve graphify for genuine `app.js`/JS-architecture questions.
- For JS-architecture questions, run `graphify query "<question>"` when graphify-out/graph.json exists. Use `graphify path "<A>" "<B>"` for relationships and `graphify explain "<concept>"` for focused concepts. These return a scoped subgraph, usually much smaller than GRAPH_REPORT.md or raw grep output.
- If graphify-out/wiki/index.md exists, use it for broad navigation instead of raw source browsing.
- Read graphify-out/GRAPH_REPORT.md only for broad architecture review or when query/path/explain do not surface enough context.
- After modifying code, run `graphify update .` to keep the graph current (AST-only, no API cost).
- **Invocation in this environment**: the `graphify` command is *not* on PATH. Call it through the saved interpreter instead — e.g. `` "$(cat graphify-out/.graphify_python)" -m graphify query "…" `` and `` "$(cat graphify-out/.graphify_python)" -m graphify update . ``. The interpreter path lives in `graphify-out/.graphify_python` (currently the pythoncore 3.14 build that has `graphifyy` installed). Note: this graph indexes **`app.js` only** (AST) — CSS selectors and HTML are *not* nodes, so pure styling/markup edits report "no code-graph topology changes."
